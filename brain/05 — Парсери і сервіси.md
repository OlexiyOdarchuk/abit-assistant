## `internal/parser/osvita/`

Парсер vstup.osvita.ua. Найбільш стабільне джерело.

### Як працює

1. **Static page**: `GET /y<рік>/r<регіон>/<uid>/<sid>/` → HTML 60-70 KB
   - Назва ВНЗ, програма, spec_code → з `.page-vnz-detail-title h{1,2}`
   - Volume (ліцензований обсяг, квоти, статистика) → з **`<table><tr><td>k</td><td>v</td></tr>`** (це знайшли в коммі `304ecd3` — раніше шукали `<b>` тегі, не спрацьовувало)
   - JS-vars (`statuses`, `rec_types`, `subjects`, `nmts`, `sub4ar`, `k4max`, `eb`, `okr`, `rk`) — з inline `<script>` через `extractJSConfig` (word-boundary regex + balanced-bracket extractor)

2. **API requests**: `POST https://vstup.osvita.ua/api/` з payload `{action: "requests", y, sid, uid, last}` → JSON `{url: "<signed_json_url>"}`. Потім `GET <signed_json_url>` → `{requests: [...], requests_subjects: {...}}`. Fan-out на 8 workers, stride = 500*8.

### Особливості

- **Warm-up**: перший POST часто повертає порожньо (cookie ще не сетнули). Тому `fetchJSONURL` з last=0 викликається до fan-out, just to seed jar.
- **Retry on 429/5xx** з expo backoff (200ms → 400ms → 800ms), max 3 спроби.
- **Cancel-on-error**: коли один worker впав з non-retriable error, `cancel()` каскадує на всіх — щоб не довбати osvita 60 секунд від решти.
- **Незалежні lane-и** (коміт `acc15e1`): кожен worker зупиняється на ВЛАСНІЙ першій порожній сторінці й НЕ обрізає інших. Раніше був cross-lane ratchet (`stopAt = min(...)`), але transient-порожня відповідь в одному lane могла тихо втратити дані іншого. Ціна — до `workers-1` зайвих порожніх fetch на справжньому хвості.

### URL-структура (deep research 2026-06-13)

`https://vstup.osvita.ua/y<рік>/r<регіон>/<uid>/<sid>/`

- `y2025` — рік кампанії; `<sid>` (остання цифра, напр. 1471029) = **ID
  програми/конкурсної пропозиції**; `<uid>` (напр. 282) = **код закладу**
  (НЕ спеціальність). Назви `sid`/`uid` у коді історичні — regex тягне дві
  цифри коректно, значення не хардкодяться.
- ⚠️ **`r<N>` — код ОБЛАСТІ. r14 = Львівська (НЕ Харків!), r21 = Харківська,
  r10 = Івано-Франківська.** Сегмент регіону парсер ігнорує (`[^/]+`), тож
  на роботу не впливає — але приклад у промптах історично «r14» = Львів.

### Browse-endpoint `/spec/` — розблоковує реверс-пошук (фіча №2)

osvita МАЄ сторінку переліку програм за фільтрами. Структуру розшифровано
на живих сторінках (2026-06-13):

```
/spec/<рівень>-<підстава>-<форма>/<регіон>-<E>-<спеціальність>-<G>-<H>-<offset>/
```

**1-а група (3 слоти) — тип навчання:**
- `рівень` — освітній ступінь: `1`=Бакалавр, далі Магістр/Мол.бакалавр/PhD…
- `підстава` — підстава вступу: `40` спостережено для нашого кейсу
  (ПЗСО/після 11). Точні коди — у `<select>` форми.
- `форма` — `1`=Денна, далі Заочна/Вечірня/Дистанційна.
- Приклад нашого кейсу (бакалавр ПЗСО денна): **`1-40-1`**.

**2-а група (6 слотів) — фільтри + пагінація.** Повну розкладку взято з
form-submit JS самого osvita (не вгадано):
`<region>-<industryId>-<specialityId>-<universityId>-<budget>-<offset>`
- `region` (слот 1) — `21`=Харків, `0`=всі. **Та сама нумерація, що `rNN`
  у URL програми** (r21=Харків і там, і там).
- `industryId` (слот 2) — галузь знань: `166`=ІТ, `164`=Бізнес/право,
  `167`=Інженерія… (11 шт., статично у `<select>` форми). Кожна мапиться 1:1
  на офіційну літеру галузі (A–K) — визначено емпірично з `/spec/` (усі
  спеціальності галузі мають один префікс, напр. F3 Комп'ютерні науки):
  161→A Освіта, 162→B Культура/мистецтво/гум., 163→C Соц./журналістика,
  164→D Бізнес/право, 165→E Природничі, 166→F ІТ, 167→G Інженерія,
  168→H Аграрні, 169→I Охорона здоров'я, 170→J Транспорт, 171→K Безпека.
  У боті — `galuzLetter` (where.go), показується в пікері галузей.
- `specialityId` (слот 3) — osvita-внутрішній ID: `2548`=Комп'ютерні науки.
  ⚠️ НЕ офіційний код (122). ⚠️ **dropdown порожній** — вантажиться AJAX-ом
  каскадно після вибору галузі (статичного джерела не знайдено; open
  question). Для фіч НЕ блокер — фільтруємо галуззю+регіоном, а спеціальність
  матчимо за назвою серед результатів.
- `universityId` (слот 4) — код ЗВО, **= середній сегмент URL програми**
  (напр. 318). `0`=всі.
- `budget` (слот 5) — **`1` = тільки бюджетні пропозиції**, `0`=всі. ⭐ для
  «куди я поступлю на бюджет».
- `offset` (слот 6) — пагінація, **+50 на сторінку**.

**JSON-довідники osvita** (статичні, без скрейпу):
- `/doc/json/universities_0.json` (~1.4 МБ) — повний довідник ЗВО:
  `{universityId, universityShortName, universityFullName}`. ✅ Завантажувач
  `Parser.FetchUniversities` + `MatchUniversity` (нормалізований матч назви).
- `/doc/json/industries.json` → `[]` (порожній), `specialitylist.json` → 404.
  Галузі беремо зі `<select>`, спеціальності — див. нижче.

**Endpoint спеціальностей (знайдено 2026-06-13):** `/api/?get=specialities`
(за патерном `/api/?get=universities`, який віддає довідник ЗВО). ⚠️ Зараз
повертає `[]` — **міжсезоння**, даних немає; невідомо, чи фільтрує параметр
`industryId`/`industry` (обидва дали `[]`). Тож вибір спеціальності ДО browse
**відкладено до старту кампанії 2026** — будувати наосліп проти порожнього
endpoint не можна. Поки що працює **post-browse фільтр спеціальності**
(`internal/bot/where.go`): мітки з результатів галузі (несуть букву-код,
напр. «F3 Комп'ютерні науки») → пікер. Це повністю закриває «вказати
конкретну спеціальність» без залежності від endpoint.

**Поведінка результату:** заголовок «Знайдено: N (спеціальності за РІК)» +
до 50 лінків програм `/yYYYY/rNN/<заклад>/<програма>/`. Приклади живих:
`/spec/1-40-1/0-0-2548-0-0-0/` → 370 програм КН; `/spec/1-40-1/21-0-0-0-0-0/`
→ всі бакалаврські програми Харкова.

**Коди регіонів і спеціальностей** — у `<select>` форми `/spec/`. План:
**скрейпити їх раз у рантаймі** (парсити option value→назва), не хардкодити —
2026 коди оновляться самі. ⚠️ Рефутнуто (0-3): NIE `/spec/N-M-K/` =
регіон-спец-рівень; не вгадувати слоти — мапити з живих даних.

**Для фічі №2:** ітерувати offset 0,50,… для (регіон, спеціальність) поки
не зберемо «Знайдено: N» → список program URL → кожен через наявний Analyze.
**Для агресивного №1:** резолвер (назва ВНЗ + спец) → відфільтрувати `/spec/`
за спеціальністю → fuzzy-match по ВНЗ → osvita URL → fetch+rank конкурента.

## `internal/parser/abitpoisk/`

Пошук конкретного абітурієнта на abit-poisk.org.ua.

### Особливості

- `POST /api/statements` з `search=<name>` → JSON `{html: "<table>...</table>"}`
- **Broken cert chain**: abit-poisk віддає incomplete TLS chain. `WithInsecureTLS()` як opt-in option. У CLI ввімкнено за замовчуванням з warning.
- `WithInsecureTLS()` тепер мутує існуючий transport, не replace client (коміт `f6192b7`) — щоб не губити CookieJar з попередніх опцій.

## `internal/service/`

Application layer. Компонує парсери + storage у use cases.

### `ProgramService`

```go
Fetch(ctx, url)   *Program       // cache-aware
Refresh(ctx, url) *Program       // bypass cache
FetchDecoded(ctx, url) []Abiturient  // Fetch + abit.Decode
```

**singleflight**: концурентні `Fetch(url)` для тієї самої URL → один реальний запит до osvita. Інші чекають його результату. Інакше viral-share = 50 паралельних парсингів. Коміт `3363e59`.

### `ApplicantService`

```go
Search(ctx, name) []ApplicantEntry  // cache-aware
Refresh(ctx, name) []ApplicantEntry // bypass cache
```

**ErrNoData НЕ кешується**. Якщо людина щойно з'явилась на abit-poisk, не хочемо 24 години показувати "не знайдено".

### `EnrichService` (поки не використовується)

```go
Enrich(ctx, abits) []EnrichedAbiturient
```

Паралельний lookup кожного абіт через ApplicantService. Worker pool capped (default 4). Призначений для майбутнього "show me the 50 most-likely competitors with their other applications".

Готовий, але не wired у handler — Telegram'у важко рендерити 50 рядків з historу одночасно. Залишимо для аналітичного режиму у server-frontend (якщо колись).

## `internal/edbo/`

Не використовується у боті зараз. Чекає [[06 — EDBO Research|EDBO]].
