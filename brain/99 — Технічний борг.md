# 99 — Технічний борг і нотатки

Те, що знаємо і не закрили. Не критично, але краще згадати ніж переоткривати.

## Не зроблене з security/correctness аудиту

### SB applicants у general pool

`analyze.go:140` — SB (співбесіда) абітурієнти потрапляють у general bucket, а у реальності вони мають свій окремий admission track. Інфлюейтить perceived competition для non-SB.

Чому не виправили: потребує доменного знання — чи SB seats виділяються окремою цифрою у `Volume`, чи це підмножина основного бюджету. Перевірити коли буде 2026 кампанія і реальний приклад з ненульовою SB-quota.

### Per-URL overrides

`handlers.go:516` — overrides зберігаються у FSM з прив'язкою до поточного URL. Якщо юзер переключається між двома програмами і повертається — overrides першої губляться.

Фікс: зробити nested map `{url: {applicant_id: bool}}` у FSM. Поки не великий impact — більшість юзерів дивляться одну програму за раз.

### Bandwidth bound на osvita output

`internal/parser/osvita/osvita.go::fanOut` — нема upper-bound на `len(requests)`. Теоретично runaway osvita response = OOM. Реально 2278 заяв = 8 МБ JSON, ОК. Якби osvita зламалась і повернула 1M — питання.

Фікс: clamp на 100k requests з explicit error.

### Cache-row DisallowUnknownFields

`storage.go:332` — JSON decode без strict mode. Не критично доки оператор єдиний has access до DB.

## Не зроблене з UX-аудиту

### `/admin` broadcast Markdown validation

Поточно: admin вводить текст з Markdown, preview рендериться. Якщо `*` непарний — preview показує помилку, але runBroadcast все одно стартує і кожне з 2000 повідомлень повертає 400.

Фікс: тест-send у admin chat перед стартом. Якщо 400 — переривати з підказкою. Не зробили бо broadcast рідко використовується, а admin зазвичай знає що пише.

### Histogram "you are here" line

`internal/visualizer/histogram.go` — кольори бакетів відносно userScore, але немає вертикальної лінії "ось ти". Очевидно з кольорів, але візуально менш чітко.

`go-chart/v2` annotation API нетривіальний — відкладено.

Колись зробити `Makefile` зі стандартними цілями (`make test`, `make build`, `make docker-build`). Поки `go test ./...` достатньо.

Не міряємо `go test -cover`. Точно <80% — handlers малопокриті. Бот складно мокати без reverse engineering telebot internals.

## Нотатки на майбутнє

### sqlc rename — нова field?

Якщо додаєш field `tg_id`-стилю до query → додай у `sqlc.yaml::rename`. Інакше згенерується `TgId` замість `TgID`.

### Migration robustness

Поточний `applyMigrations` — forward-only, transactional per file. Якщо посеред міграції SIGKILL — частково applied. На SQLite це окей бо все в одній транзакції, але треба уважно якщо ADD COLUMN + UPDATE backfill в одному файлі (як 0003).

### Telegram API rate

Bot використовує long-polling, не webhook. 30 updates/sec ліміт нас не зачіпає (юзерів ще не настільки багато). Broadcast лімітований нами явно 50ms/повідомлення (~20/sec). Якщо userbase виросте 10k+ — переключитись на webhook та черги.

### Backup стратегія

Поки нема. `data/abit.db` єдиний source of truth. У Docker compose бі hostmount — backup через `cp data/abit.db data/abit.db.$(date +%F)` плюс copy off-host.

Колись додати `aa backup` CLI команду.
