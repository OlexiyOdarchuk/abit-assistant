# 07 — Розробка

## Підготовка

- Go 1.26.3+
- (опц.) `sqlc` через `go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest`
- (опц.) Docker якщо хочеш деплоїти через compose

```bash
git clone git@github.com:OlexiyOdarchuk/abit-assistant.git
cd abit-assistant
cp .env.example .env
# відредагуй .env: TELEGRAM_TOKEN, опц. ADMIN_IDS
```

## Запуск локально

```bash
go run ./cmd/bot
# або
CGO_ENABLED=0 go build -trimpath -ldflags="-s -w" -o aa-bot ./cmd/bot
./aa-bot
```

База даних резолвиться через [[04 — Зберігання#Шлях до БД]]. Стартовий лог показує абсолютний шлях + кількість юзерів — якщо `users: 0` після рестарту, значить новий файл (хибний cwd чи зачистили).

## Запуск у Docker

```bash
mkdir -p data && chmod 777 data    # UID 65532 у контейнері
docker compose up -d
docker compose logs -f
```

`data/` лишається на хості — бот переживе `docker compose down`. Образ scratch, ~13 МБ.

## Dev CLI

`cmd/cli/main.go`:

```bash
go run ./cmd/cli osvita https://vstup.osvita.ua/y2025/r14/282/1471029/
go run ./cmd/cli abitpoisk "Куцелюк Д О"
go run ./cmd/cli osvita ... | go run ./cmd/cli decode
go run ./cmd/cli edbo decrypt <base64> <salt> <year>
```

Корисно коли треба швидко перевірити чи парсер працює, без бота.

## Тести

```bash
go test ./...              # всі
go test ./pkg/abit/...     # доменна логіка
go test -v ./internal/bot/ # бот (smoke + регрес від reviewу)
go vet ./...               # static checks
```

Всі тести зелені на коміт `9b9f58b`. ~80 тестів.

## sqlc — коли треба

Якщо змінив `internal/storage/queries/*.sql` або `migrations/*.sql`:

```bash
~/go/bin/sqlc generate
```

Згенерує заново `internal/storage/db/*.go`. **НЕ редагувати руками** ці файли.

## Структура імпортів

```
pkg/abit               — никуда не імпортує внутрішнього
pkg/parser/*           — імпортує pkg/abit
internal/*             — імпортує pkg/* + сусідів за дозволом

cmd/bot/main.go        — імпортує все, тримає graph
cmd/cli/main.go        — імпортує pkg + internal/{edbo, config}
```

Циклічних залежностей нема. `go vet` стежить.

## Env

| Змінна | Default | Що |
|---|---|---|
| `TELEGRAM_TOKEN` | — (required) | Bot API token від @BotFather |
| `ADMIN_IDS` | `""` | comma-separated tg_id для `/admin` |
| `DATABASE_PATH` | XDG (див. вище) | Шлях до SQLite файлу або `:memory:` |
| `LOG_LEVEL` | `info` | debug / info / warn / error |

Все читається через `internal/config`, який сам завантажує `.env` якщо знайде.

## Логи

JSON через `log/slog`. У compose дивитись `docker compose logs -f`. Локально — stderr.

Кожен handler логується `handler ok` (debug) або `handler failed` (error) з `user_id`, `took_ms`, `text`.

Помилки що містять `sql:` / `panic:` / `/home/` / etc. — колапсуються у генеричне "сталася внутрішня помилка" для юзера, повний текст лишається в логу (див. `userFacing` у middleware.go).

## Релізи

Поки що тегів нема. Якщо колись треба:
```bash
git tag -a v3.0.0 -m "first stable Go release"
docker build --build-arg VERSION=v3.0.0 -t abit-assistant:v3.0.0 .
```

`VERSION` ldflag вбудовується в бінарник через `-X main.version=...`. Зараз `main.go` не показує його ніде — додати у `/about` коли реліз буде.

## Pitfalls для майбутнього себе

1. **`./data/abit.db` різний при різних cwd** → дані "губляться" → див. [[04 — Зберігання#Шлях до БД]]
2. **Telegram callback_data ≤ 64 байти** → не клади туди URL → FSM `state.Data[fsmKeyURL]` замість того
3. **Telegram legacy Markdown** ламається на `]` / `(` / `)` у user-text → завжди `mdEscape`
4. **`go test ./...` не запускає `cmd/cli` чи `cmd/bot`** — там немає тестів. Це нормально
5. **sqlc rename**: `tg_id` → `TgID`, `url` → `URL`. Якщо новий field — додати в `sqlc.yaml::rename`
6. **`recoverPanics` має бути INNERMOST** middleware → див. [[03 — UX і Bot#Middleware chain]]
